# MVP와 사업 검증

## MVP 목표

첫 제품은 “AI가 봇을 완벽히 알아낸다”가 아니라 다음 가설을 증명해야 합니다.

> 2GB급 VPS에서 정상 사용자의 응답속도를 거의 훼손하지 않으면서, 서버 자원을 고갈시키는 자동화 트래픽을 설명 가능한 근거로 제한하고 필요할 때만 Cloudflare 방어로 전환할 수 있는가?

## 개발 배치

### 0. 기존 자산 복구

- 월척웹 `edge_proxy` 소스와 테스트 복구
- 새 저장소와 라이선스 결정
- Pingora 의존성 고정
- 기존 기능·보안·품질 감사
- GnuBoard 테스트 서버에서 loopback smoke

통과 조건: 기존 프록시 기능이 재현되고 제거 당시의 오래된 운영 설정이 새 제품에 섞이지 않아야 합니다.

### 1. 관찰 전용 edge

- Pingora 직접 80/443
- 인증서 로드·갱신 hook
- Nginx 또는 Apache loopback upstream
- 구조화된 요청·응답 메트릭
- PHP-FPM·MySQL·Redis·OS 수집
- 차단 없이 사건 리포트만 생성

통과 조건: 직접 Nginx 대비 정상 요청의 추가 지연과 오류율이 허용 범위 이내여야 합니다.

### 2. 로컬 보호

- 경로별 비용표
- IP·세션별 rate limit
- 고비용 기능 보호
- IP·CIDR 임시 차단
- 검증 쿠키와 challenge
- 관리자의 즉시 해제

통과 조건: 사람 시나리오 오탐 없이 봇 시나리오가 PHP와 DB에 도달하기 전에 제한돼야 합니다.

### 3. 비상 전환

- Cloudflare 최소 권한 API 토큰
- DNS only와 proxied 전환
- 원본 IP 직접 접근 방어
- 전환 실패와 부분 성공 감지
- 안정화 후 자동 복구
- 수동 비상 중지

통과 조건: API 성공 표시가 아니라 실제 DNS, HTTPS, 원본 방화벽과 사용자 요청 경로까지 검증해야 합니다.

### 4. 운영 제품화

- 초보자용 웹 UI
- 설치·업데이트·제거와 bypass
- 사건 타임라인과 절감 리포트
- GnuBoard·WordPress 프로필
- x86_64·aarch64 배포
- 서명된 바이너리와 SBOM

## 시험 시나리오

- k6 정상 사용자 10·20·50·100·200 VU 단계 증가
- 정적 파일을 포함하는 실제 브라우저 흐름
- 검색·깊은 페이지·회원 프로필 순회 봇
- 쿠키 없이 HTML만 수집하는 봇
- 여러 IP로 분산된 저속 고비용 요청
- 검증된 Googlebot과 위조 Googlebot
- PHP-FPM 포화, MySQL 지연과 Redis 장애 주입
- Cloudflare API 실패, DNS 전환 지연과 인증서 갱신 실패
- edge·control 재시작과 Nginx bypass

공개 서버에 실제 공격을 발생시키지 않고 격리된 테스트 서버와 재생 가능한 트래픽으로 검증합니다.

## 핵심 지표

| 지표 | 의미 |
|---|---|
| 정상 트래픽 추가 지연 | 가드 자체가 만드는 성능 비용 |
| 탐지까지 걸린 시간 | 자원 포화 전에 대응 가능한가 |
| PHP·DB에 도달하지 않은 고비용 요청 | 실제로 절감한 서버 작업량 |
| 정상 세션 오탐률 | 제품을 계속 켜둘 수 있는가 |
| 자동 전환·복구 성공률 | 야간에도 맡길 수 있는가 |
| 운영자 개입 시간 | 관리 비용을 얼마나 줄였는가 |
| 회피한 증설 추정액 | 고객에게 금전 가치를 설명할 수 있는가 |

수치는 사전 추정으로 확정하지 않고 테스트 서버와 파일럿 사이트의 기준선을 통해 설정합니다.

## 상품 포지셔닝

### 경쟁 문장

> 상시 CDN이 아니라, 작은 VPS가 위험할 때만 켜지는 자원 인식형 방어 게이트웨이

### 차별점

- 일반 RPS가 아니라 PHP·DB가 실제로 받은 비용을 판단
- Cloudflare 상시 프록시 없이 평시 직접 연결
- 공격이 아닌 AI·검색·스크래핑 봇까지 보호
- GnuBoard·WordPress 경로를 이해하는 비용 프로필
- 왜 차단했는지 사람이 이해할 수 있는 사건 리포트
- 원클릭 설치보다 안전한 자동 복구에 집중

## 수익모델 후보

| 제품 | 포함 기능 |
|---|---|
| Community | 관찰, 기본 로컬 제한, 수동 리포트 |
| Pro | 자동 Cloudflare 전환, 방화벽 연동, 앱 프로필, 장기 리포트 |
| Managed | 다중 서버, 알림, 정책 피드와 원격 운영 지원 |

봇 특성과 provider API는 계속 변하므로 완전한 일회성 판매보다 업데이트와 정책 피드를 포함한 연간 또는 월간 모델이 자연스럽습니다. 라이선스는 기존 G7 Installer의 MIT 결정을 자동으로 따르지 않고 새 프로젝트에서 별도로 결정합니다.

## 성공 가능 포인트

- AI 크롤러 증가로 작은 사이트도 자동화 트래픽 비용을 체감합니다.
- 기존 제품은 상시 프록시 또는 대형 사이트 중심인 경우가 많습니다.
- 운영자가 이해하기 쉬운 “느려지면 필요한 만큼만 보호”라는 가치가 명확합니다.
- 기존 Pingora 코드와 실제 VPS·GnuBoard 부하 테스트 경험을 재사용할 수 있습니다.
- 비용 절감 결과를 수치로 보여주면 보안 공포가 아니라 투자 대비 효과로 판매할 수 있습니다.

## 실패 가능 포인트

- 정상 사용자나 검색봇을 차단하면 제품 신뢰를 즉시 잃습니다.
- edge가 TLS를 소유하므로 인증서와 장애 책임이 커집니다.
- Cloudflare 전환 전에 원본 회선이 포화되면 로컬 대응만으로 복구하기 어렵습니다.
- VPS와 DNS 사업자마다 API와 방화벽 기능이 다릅니다.
- 봇도 JavaScript와 정상 브라우저를 사용하므로 완벽한 구분은 불가능합니다.
- 너무 많은 범용 기능을 넣으면 소형 VPS 문제에 집중한 장점이 사라집니다.
- 무료 사용자 지원 비용이 제품 수익을 초과할 수 있습니다.

## 중단 기준

다음 중 하나라도 해결되지 않으면 자동 차단 제품으로 공개하지 않습니다.

- 정상 트래픽 오탐을 안전한 수준으로 낮추지 못함
- edge 장애 시 기존 웹서버로 복구하는 절차가 불안정함
- 인증서 갱신 실패를 사전에 감지·복구하지 못함
- 정상 요청 추가 지연이 고객이 체감할 수준으로 큼
- Cloudflare 전환과 원본 방화벽 상태를 일관되게 검증하지 못함
- 절감한 비용이나 장애 감소를 고객에게 설명할 근거가 없음

## 다음 결정

1. 새 프로젝트 이름과 저장소를 확정합니다.
2. 월척웹 Pingora 코드의 마지막 정상 커밋을 기준선으로 선택합니다.
3. 직접 80/443 종단을 우선할지 Apache/Nginx 뒤 shadow 검증부터 할지 결정합니다.
4. 첫 파일럿은 `g7devops.com` 같은 격리된 테스트 서버로 제한합니다.
5. 오픈소스 범위와 유료 제어·정책 기능의 경계를 확정합니다.
