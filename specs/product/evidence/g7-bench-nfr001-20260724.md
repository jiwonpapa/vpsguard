# NFR-001 2GB Nginx 비교 성능 증거

## 결론

`NFR-001`은 격리 Ubuntu 24.04 x86_64 VM에서 통과했습니다.

- direct Nginx median p95: `0.995ms`
- VPSGuard 경유 median p95: `2.183ms`
- p95 추가 지연: `1.188ms` (`2ms` 이하)
- 처리량 감소: `0.47%` (`10%` 이하)
- direct·guard 실패율: `0`
- guard-edge RSS 관측 최대: `15,876KiB`

측정 뒤 임시 Edge·Nginx listener와 host/guest stage를 제거했고, Nginx
configuration hash·service 상태·public HTTPS 상태와 libvirt memory를 시작값으로
원복해 일치를 확인했습니다. `g7devops.com` 운영 서버에는 배포하지 않았습니다.

## 측정 계약

- 같은 VM의 private Nginx `:18081`과 `guard-edge :18080 -> Nginx :18081` 비교
- `50 VU`, `15초`, `100ms` think time, 3회 paired measurement
- 순서 편향을 줄이기 위해 `direct→guard`, `guard→direct`, `direct→guard` 교차
- 고정 HTML 경로 `/__vpsguard_nfr001`로 proxy 추가 비용을 PHP-FPM 변동과 분리
- 실제 G7 `/`는 direct·guard 양쪽 `200` smoke를 같은 실행에서 확인
- 세 round의 p95와 RPS median, 어느 한 round라도 실패율이 있으면 실패

## 방법 보정 기록

예산을 바꾸지 않고 workload 오류를 두 단계에서 제거했습니다.

1. 실제 dynamic `/`를 무한 반복한 첫 측정은 direct 자체 p95가 약 `669ms`로
   PHP-FPM 포화 변동이 proxy 비용보다 커서 p95 차이 `7.934ms`를 수용 증거로
   사용하지 않았습니다. 처리량 감소는 `1.15%`였습니다.
2. think time 없는 정적 포화 측정은 direct Nginx가 약 `95k RPS`, Edge가 약
   `27.6k RPS`여서 p95 추가 지연은 `1.742ms`지만 처리량 감소가 `71.07%`였습니다.
   이는 “정상 browse”가 아니라 최대 처리량 포화 시험이므로 수용 증거에서
   제외하고 별도 capacity 기준 후보로 남겼습니다.
3. 최종 계약은 50 VU마다 `100ms` think time을 고정해 같은 약 `500 RPS`
   제공 부하에서 proxy 지연과 처리량을 비교했습니다.

## 자동 원복

하네스는 성공·실패 모두에서 다음 순서로 원복합니다.

1. `/proc/<pid>/exe`가 전용 stage의 `vps-guard-edge`인지 확인
2. `TERM`, bounded wait, 필요할 때만 같은 executable PID에 `KILL`
3. 전용 Nginx config 하나만 제거하고 `nginx -t` 뒤 reload
4. libvirt memory와 guest MemTotal을 시작값으로 복구
5. host·guest의 commit 전용 stage만 제거
6. Nginx full configuration hash, service, public HTTPS와 memory exact read-back

원본 site data, PHP-FPM socket, TLS 인증서와 public 80/443 server block은
수정 대상에 포함하지 않았습니다.
