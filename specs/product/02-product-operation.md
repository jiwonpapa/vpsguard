# 제품 동작

## 운영 원칙

VPS Guard는 곧바로 차단하지 않습니다. 관찰, 로컬 제한, 사용자 검증, 외부 방어 전환 순으로 대응하며 각 판단 근거를 기록합니다.

## 방어 상태

| 코드 | 표시 | 조건 | 자동 조치 |
|---|---|---|---|
| `NORMAL` | 정상 | 지연과 자원 사용이 기준선 이내 | 원본 직접 연결, 관찰만 수행 |
| `WATCH` | 주의 | 특정 IP·세션·경로의 비용 증가 | 경고, 상세 기록, 약한 속도 제한 |
| `LOCAL_GUARD` | 로컬 방어 | 봇 가능성·자원 비용 또는 서버 압력이 지속 증가 | 고비용 경로 제한, 429, 검증, TTL 차단 |
| `EMERGENCY_PROXY` | 비상 보호 | 분산 요청 또는 로컬 방어 한계 초과 | Cloudflare 프록시 ON, 검증 후 원본 보호 |
| `RECOVERING` | 복구 | 지정 시간 동안 부하와 오류율 안정 | 단계적으로 제한 해제 후 DNS only 복귀 |
| `MANUAL_HOLD` | 수동 고정 | 관리자가 자동 전이를 중지 | 현재 안전 한도를 유지하고 외부 자동 조치 중단 |

상태를 올리는 조건과 내리는 조건을 다르게 두는 히스테리시스를 적용합니다. 짧은 부하 변동 때문에 프록시가 반복해서 켜지고 꺼지면 안 됩니다.

## 정상 요청 경로

```text
사용자
  -> Pingora Guard :80/:443
    -> Nginx 또는 Apache :loopback
      -> PHP-FPM
        -> MySQL/Redis
```

Cloudflare DNS는 사용하되 레코드는 DNS only 상태를 유지합니다. Pingora Guard는 로컬 VPS에서 요청을 통과시키므로 해외 프록시 왕복을 추가하지 않습니다.

## 비상 전환 순서

1. 공격 주체에 로컬 속도 제한 또는 차단을 즉시 적용합니다.
2. Cloudflare API로 서비스 레코드를 프록시 상태로 변경합니다.
3. 프록시 경유가 확인되면 원본의 80/443 접근을 Cloudflare 대역 중심으로 제한합니다.
4. SSH 포트와 현재 관리 세션은 절대 자동 차단하지 않습니다.
5. 상태, 조치, API 응답과 실패 원인을 사건 타임라인에 기록합니다.
6. 안정화 시간이 충족되면 원본 방화벽과 DNS를 검증된 순서로 복구합니다.

공격자가 원본 IP를 이미 알고 있을 수 있으므로 프록시 전환만으로 완료 처리하지 않습니다. 로컬 차단과 원본 보호가 함께 동작해야 합니다.

## 사용자 검증

의심 트래픽 전체에 CAPTCHA를 보여주지 않습니다.

- 정상 세션: 즉시 통과
- 약한 의심: 서명된 단기 쿠키와 가벼운 JavaScript 검증
- 높은 의심: 검증 페이지 또는 선택형 Turnstile 연동
- 브라우저가 아닌 공개 API: API 키·서명·별도 한도 사용
- 검증된 검색봇: 크롤링은 허용하되 고비용 경로는 별도 제한

## 기능별 보호 모드

사이트 전체를 닫기 전에 비용이 큰 기능부터 보호합니다.

- 검색 일시 제한
- 깊은 페이지네이션 제한
- 로그인·비밀번호 찾기 속도 제한
- 회원 프로필 대량 순회 제한
- 업로드 동시 실행 제한
- 캐시 미스가 큰 피드·목록 보호
- 정적 파일과 이미 캐시된 공개 페이지는 계속 제공

## 웹 관리 화면

관리 화면은 보안 용어보다 운영 결과를 먼저 보여줍니다.

- 현재 상태: 정상·주의·로컬 방어·비상 보호·복구·수동 고정
- 왜 이 상태인지 설명하는 상위 근거 3개
- 보호 전후 CPU, PHP-FPM, DB 지연과 응답시간
- 차단·제한·검증 중인 주체와 만료시간
- Cloudflare와 원본 방화벽의 실제 적용 상태
- `자동 대응 중지`, `비상 보호 시작`, `정상 모드 복구` 명령
- 사건별 타임라인과 내려받을 수 있는 진단 리포트

자동 조치는 관리자가 언제든 중단할 수 있어야 하며 위험한 복구 명령에는 영향 범위와 확인 절차를 제공합니다.
