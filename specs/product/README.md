# VPS Guard 아이디어 북

> 상태: 초기 구현 SDD
> 작성 기준일: 2026-07-14
> 구현 위치: `g7-installer`와 분리할 별도 프로젝트

이 폴더는 단순 아이디어 메모가 아니라 새 프로젝트를 시작하기 위한 초기 스펙 주도 개발 문서 세트입니다. 별도 초안 SDD를 다시 만들지 않고 이 문서군을 새 저장소로 옮겨 구현을 시작합니다.

## 정본 순서

1. 구현 전에는 [MASTER SDD](MASTER_SDD.md)와 요구사항 ID가 계약의 정본입니다.
2. 구현 후에는 Rust 타입, module rustdoc, 설정 스키마와 상태 전이 코드가 실행 정본입니다.
3. 테스트 추적표는 요구사항 ID와 코드·테스트·운영 증거를 연결합니다.
4. 사업 설명이나 아이디어가 구현 계약과 충돌하면 MASTER SDD가 우선합니다.

문서에서 `필수`는 구현과 테스트가 없으면 완료할 수 없다는 뜻이고, `금지`는 예외 없이 차단해야 한다는 뜻이며, `권장`은 근거를 기록하면 변경할 수 있다는 뜻입니다.

## 한 문장 정의

평상시에는 VPS 원본 서버의 빠른 직접 연결을 유지하고, 자원을 고갈시키는 봇이나 이상 트래픽이 발생할 때만 로컬 방어와 Cloudflare 프록시를 단계적으로 가동하는 소규모 사이트용 적응형 보안 게이트웨이입니다.

## 가장 중요한 문제

소규모 사이트 운영자는 현재 다음 두 선택지 사이에 놓입니다.

1. 원본 서버에 직접 연결해 빠르게 서비스하지만 봇과 공격에 노출됩니다.
2. Cloudflare 프록시를 상시 사용해 보호받지만 국내 사용자의 체감 성능을 희생합니다.

VPS Guard는 세 번째 선택지를 제공합니다.

> **평소에는 빠르게, 위험할 때만 보호받고, 안정되면 자동으로 원상 복귀합니다.**

## 대화에서 확정한 전제

- 국내 사용자를 대상으로 Cloudflare 프록시를 상시 켜면 체감 성능이 나빠진다는 운영 경험을 제품 전제로 사용합니다.
- DDoS뿐 아니라 검색봇, AI 학습봇, AI 검색봇, 스크래퍼처럼 서버 자원을 소모하는 자동화 트래픽 전체를 다룹니다.
- 단순 요청 수가 아니라 PHP-FPM, DB, Redis와 고비용 URL에 미치는 영향을 판단합니다.
- 정상 상태에서는 Cloudflare DNS only를 유지합니다.
- Cloudflare 프록시는 비상 방어 수단이며 안정화 후 자동 해제합니다.
- 최앞단 데이터 플레인은 기존 월척웹 Pingora 프록시 자산을 재사용합니다.
- 설치 도구와 유지보수·방어 도구는 분리합니다.
- 첫 검증 대상은 1~4GB VPS에서 운영하는 GnuBoard와 WordPress 계열 사이트입니다.

## 제품 경계

VPS Guard는 CDN, 범용 웹서버, 서버 설치기, 애플리케이션 튜너를 새로 만드는 프로젝트가 아닙니다. 기존 Nginx 또는 Apache와 애플리케이션 앞에서 트래픽을 판정하고, 서버 상태를 수집하고, 필요한 방어 조치를 자동화하는 제품입니다.

새 저장소를 만들 때 이 폴더를 `specs/product/`로 옮기고 `MASTER_SDD.md`를 초기 제품 계약으로 사용합니다. 별도 `SPEC.md`를 다시 작성해 내용이 갈라지게 하지 않습니다.
